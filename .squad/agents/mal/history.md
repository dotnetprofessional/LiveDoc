- **2025-07-25**: Analyzed scenario-level attachment gallery feasibility. Key finding: the schema (`Node.execution: ExecutionResult.attachments`) already supports attachments at every node level including Scenario — it's just never populated above step level. Client-side aggregation via `flatMap` over `steps[].execution.attachments` is the right approach for Phase 1 — zero backend/schema changes needed. AttachmentViewer already handles multi-attachment browsing with prev/next. Phase 1 is ~50-80 lines across 3 files (ScenarioBlock, AttachmentViewer, utility function). Scenario Outline support deferred to Phase 2 due to per-example-row aggregation complexity. **Implemented**: Kaylee completed full scenario-level gallery with 170-line utilities (GalleryItem, StepGroup, navigation helpers), enhanced AttachmentViewer with step context bar, auto-play, keyboard nav, film strip dividers, and integrated into ScenarioBlock + StepList for unified step-level entry. Architecture validated as backward-compatible; no breaking changes.
- **2025-07-29**: Documented JSON export + static HTML export across all six doc pages. Key doc architecture: `export` option on `LiveDocSpecReporter` (TS) and `LIVEDOC_EXPORT_PATH` env var (.NET) both produce TestRunV3 JSON; `livedoc-viewer export` CLI converts to self-contained HTML. Docs structure: reporters.mdx (reference), ci-cd.mdx (guide, now primary pipeline uses `export` not `JsonReporter`), configuration.mdx (xUnit reference), cli-options.mdx (viewer CLI reference with export subcommand), static-export.mdx (new guide — full pipeline walkthrough). All cross-referenced. The `export` option writes alongside server publishing, not instead of. `LiveDocTestRunReporter.IsEnabled` checks both serverUrl and exportPath.
- **2025-07-30**: Reviewed Playwright integration bug fixes. Key architecture: `onScenarioStart`/`onScenarioEnd` hooks in `livedoc.ts` (module-level arrays, lines 96-97) fire in `beforeAll`/`afterAll` of each scenario's `describe` block — correct placement since LiveDoc scenarios are vitest describes, not tests. Hooks fire in both `scenario()` and `scenarioOutline()` example blocks. The `useBrowser()` function in `packages/vitest/_src/app/playwright/index.ts` cleanly branches between shared context (one context for all scenarios) and fresh context (per-scenario isolation). Bug 2 fix: `browser.close()` in outer `afterAll` is sufficient — Playwright cascades closure to all contexts/pages. Identified concern: `scenarioEndHooks` loop has no error isolation — one failing hook blocks subsequent hooks and afterBackground cleanup.
- **2025-01-XX**: Designed comprehensive E2E test strategy to prevent regressions from reaching consumers. **Root cause of recent regressions**: Unit tests validate components in isolation, but no tests validate the full pipeline (Spec → Reporter → JSON → Server → Viewer → UI). **Strategy**: 4 test categories across new `packages/e2e/` package: (1) Reporter output schema validation (run fixture specs, validate JSON against Zod schema), (2) Viewer rendering validation (seed viewer with synthetic JSON via HTTP POST, use Playwright to validate UI correctness — includes critical breadcrumb regression test), (3) Full pipeline integration (orchestrate server + viewer + vitest, validate end-to-end flow), (4) Bundle integrity (already exists in `bundled-output-integrity.Spec.ts`). **Key architectural decisions**: Single E2E package (not distributed), synthetic JSON fixtures for viewer tests (fast + controlled), real spec fixtures for pipeline tests (validates integration), HTTP POST seeding via `/api/runs/publish` (matches real-world usage), Playwright for UI validation (already a peer dep), dynamic ports via `get-port` (parallel-safe). **Implementation priority**: Phase 1 = breadcrumb regression test (1-2 days, highest ROI), Phase 2 = full viewer rendering suite (2-3 days), Phase 3 = reporter validation (2 days), Phase 4 = pipeline tests (2-3 days), Phase 5 = CI integration (1 day). **Target performance**: < 2 minutes total E2E runtime. Strategy document: `.squad/decisions/inbox/mal-e2e-test-strategy.md`
