## See the Difference

Most tests verify behavior but document nothing. LiveDoc tests do both:

<div className="comparison-grid">

<div className="comparison-card comparison-before">

**Typical test** — verifies, but tells the reader nothing:

```
test('discount', () => {
  const result = applyDiscount(100, 'SAVE20');
  expect(result).toBe(80);
});
```

<span className="comparison-label">✗ What's being tested? Why 80? What's SAVE20?</span>

</div>

<div className="comparison-card comparison-after">

**LiveDoc spec** — verifies *and* documents:

```
Scenario: Applying a discount code
  Given the cart total is '$100.00'
  When the user applies code 'SAVE20'
  Then the total becomes '$80.00'
```

<span className="comparison-label">✓ Self-documenting. Values in titles. No drift.</span>

</div>

</div>

The LiveDoc spec on the right executes as a real test *and* generates browsable documentation. When the test changes, the docs update automatically. When the test passes, the documentation is accurate.

## How It Works

<div className="workflow-steps">

<div className="workflow-step">
<div className="workflow-number">1</div>
<h3>Write Specs</h3>
<p>Author BDD features or specification rules in TypeScript or C#. Your tests are the single source of truth.</p>
</div>

<div className="workflow-connector">→</div>

<div className="workflow-step">
<div className="workflow-number">2</div>
<h3>Run Tests</h3>
<p>Execute with Vitest or xUnit. LiveDoc captures titles, values, steps, and results into a structured reporting model.</p>
</div>

<div className="workflow-connector">→</div>

<div className="workflow-step">
<div className="workflow-number">3</div>
<h3>Browse Docs</h3>
<p>The Viewer renders your test results as browsable, hierarchical specifications — filterable by tags, status, and search.</p>
</div>
